## 1. Purpose

Deliver execution-grade parity for inference-focused model coverage between Python
`mlx-lm` and Ruby `mlx-ruby-lm`, with reproducible dependency pins and measurable
acceptance gates.

This revision removes ambiguity that blocked prior execution:
- exact inventory freeze (no approximate counts)
- explicit submodule pin gate for ONNX
- phase boundaries that match actual dependencies
- CI/test tiering for full-matrix coverage

## 3. Scope

### In Scope

- Shared model infrastructure for parity (RoPE variants, cache families, MLA, SSM,
  gated-delta primitives, switch-layer extensions, bitlinear support)
- Architecture ports from Python model files to Ruby model files + registry
- HuggingFace Hub inference workflows (download/cache/load/save/upload/manage)
- ONNX compatibility/export validation for in-scope architecture set

### Out of Scope

- Training stack (datasets, trainer, callbacks, losses)
- Speculative decoding
- Advanced quantization research paths (AWQ, GPTQ, DWQ)
- lm-eval integration
- GGUF export

## 5. Public API Contract Additions

### `load(...)` model resolution order

`LoadUtils.load` must resolve inputs in this exact order:
1. Existing local filesystem path
2. HuggingFace repo id
3. Cache reuse policy for repo id + revision

Errors must include resolution attempts (local path, repo id, revision).

### ONNX validation artifact contract

Every ONNX model test result must include:
1. compatibility report
2. full export outcome
3. unsupported-op set (if any)
4. dependency metadata (`mlx-ruby` and `mlx-onnx` SHAs)

## 6. Revised Phase Plan

## Phase 0: Governance + Gates (Execution bootstrap)

**Objective:** Make planning artifacts enforceable.

**Deliverables:**
- inventory generator script
- committed inventory snapshot
- minimum `mlx-onnx` SHA test gate
- inventory freshness test gate

**Exit Criteria:**
- `bundle exec rake test:parity` includes governance gates and they pass.

## Phase 1: Shared Infrastructure

**Objective:** Build shared building blocks used by downstream model phases.

### 1A RoPE variants
- `SuScaledRoPE`, `Llama3RoPE`, `YarnRoPE`, `initialize_rope(...)`

### 1B Cache extensions
- `_BaseCache`, `ArraysCache`, `CacheList`, `QuantizedKVCache`, `ChunkedKVCache`

### 1C MLA
- `MultiLinear`, `QuantizedMultiLinear`, quantization conversion support

### 1D SSM primitives
- `compute_dt`, `segsum`, `ssm_attn`, `ssm_update`

### 1E Misc shared blocks
- `gated_delta`, `SwitchMLP`, `QuantizedSwitchLinear`, `PipelineMixin`,
  `BitLinear`, `XieLU`

**Exit Criteria:**
- infra parity tests pass and capability matrix is updated.

## Phase 2A: Pure Dense Transformers

**Objective:** Port dense models requiring only baseline attention/caching.

**Exit Criteria:**
- per-model forward parity
- ONNX compat + full export for this subset

## Phase 2B: Dense + Specialized Attention/RoPE

**Objective:** Port dense models with specialized RoPE/attention behavior.

**Exit Criteria:**
- same as 2A plus explicit dependency checks for specialized features

## Phase 3: MoE Models

**Objective:** Port MoE families using switch layers and router behavior.

**Exit Criteria:**
- forward parity
- routing parity
- ONNX compat coverage with documented full-export exceptions

## Phase 4: DeepSeek Family

**Objective:** Port DeepSeek V2/V3/V3.2 and related variants.

**Exit Criteria:**
- forward parity
- MLA parity
- routing parity

## Phase 5: SSM + Hybrid Models

**Objective:** Port Mamba/RWKV/hybrid families requiring SSM state paths.

**Exit Criteria:**
- forward parity
- single-step + multi-step state parity
- mixed cache parity (KV + Arrays)

## Phase 6: Gated-Delta Families

**Objective:** Port gated-delta-based models and wrappers.

**Exit Criteria:**
- forward parity
- state update parity

## Phase 7: Long-Tail Specialized Models

**Objective:** Port remaining specialized architectures and wrappers.

**Exit Criteria:**
- forward parity
- model-specific parity checks (for example bitlinear/phi3small requirements)

## Phase 8: HuggingFace Hub Integration

**Objective:** Full inference-oriented HF lifecycle support.

**Required semantics:**
- retry/backoff for network errors
- deterministic cache pathing
- cache validation and stale handling
- auth precedence (`HF_TOKEN` env first, cache second)
- save/shard/upload workflows

**Exit Criteria:**
- end-to-end load/save/upload tests pass with cache coverage

## Phase 9: ONNX Full Validation Matrix

**Objective:** cover all in-scope architectures with compatibility and export
reporting.

**Prerequisites:**
- G2 passes (minimum `mlx-onnx` SHA gate)
- relevant phase model tests are green

### Execution tiers
- PR smoke matrix: small deterministic subset
- Nightly full matrix: all in-scope architectures
- Release gate: full matrix green vs previous baseline

**Exit Criteria:**
- full matrix executed with artifacted reports
- remaining unsupported ops and crashes tracked with owners
